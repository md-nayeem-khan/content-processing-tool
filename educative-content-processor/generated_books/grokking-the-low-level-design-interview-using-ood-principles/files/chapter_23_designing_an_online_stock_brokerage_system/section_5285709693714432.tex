\subsection{Design highlights}\label{vzS1H8GD_b9PrECX8OaTR}
\\
The online stock brokerage system is thoughtfully architected, employing best practices for clarity, maintainability, and scalability. Below are key aspects of the design:

\subsubsection{Design approach}\label{Ly5RtGBa9D5UknPbmvZY1}
\\
The system is designed using a bottom-up methodology. Core components and domain entities (such as stocks, orders, and accounts) are identified first, and higher-level workflows and relationships are built on these foundations.

\subsubsection{Design principles}\label{FfQNanqMJ4Hf8KZZSQVNG}
\\
The system applies core SOLID principles, ensuring a robust and extensible design:

\begin{itemize}
\item
\protect\phantomsection\label{gfaqzLmdvUwQVnvLkTYSG}
\textbf{Single Responsibility principle (SRP):} Each class (e.g., \texttt{Order}, \texttt{Notification}, \texttt{StockInventory}) is responsible for a single aspect of the system, such as order management, notifications, or stock data.
\item
\protect\phantomsection\label{3j89iXc8bfq91YcsQEGoj}
\textbf{Open/Closed principle (OCP):} The order and notification hierarchies are open for extension (new order or notification types can be added) but closed for modification of existing code.
\item
\protect\phantomsection\label{uTr_d4-sTzzR5PHxW_-We}
\textbf{Liskov Substitution principle (LSP):} Subclasses such as \texttt{MarketOrder}, \texttt{LimitOrder}, etc., can be used interchangeably via the \texttt{Order} base class.
\item
\protect\phantomsection\label{5prI5sxAeh5YTROIJpZTV}
\textbf{Interface Segregation principle (ISP):} \texttt{StockInventory} implements the \texttt{Search} interface, ensuring classes expose only relevant behavior.
\item
\protect\phantomsection\label{PexPSazz-0LBdN3LD1jux}
\textbf{Dependency Inversion principle (DIP):} High-level modules interact with abstractions (e.g., \texttt{Order}, \texttt{Notification}) rather than concrete implementations.
\end{itemize}

\subsubsection{Design patterns}\label{TQ2PkvirYm4pmg-iTSis4}
\\
The design makes use of classic object-oriented design patterns for extensibility and clarity:

\begin{itemize}
\item
\protect\phantomsection\label{kMVpgFRwx3l7-ut_W5-i6}
\textbf{Factory pattern:} Used for creating various orders and notifications without exposing instantiation logic to the client.
\item
\protect\phantomsection\label{Vf9d89DGyWcKexO5bI2GL}
\textbf{Observer pattern:} This pattern sends members real-time notifications (via \texttt{Notification}) when trades or transactions occur.
\item
\protect\phantomsection\label{9u830rBAhXksxnyIVDm6t}
\textbf{Strategy pattern:} Applied to implement different order types, allowing interchangeable order creation and execution strategies.
\item
\protect\phantomsection\label{n7pGeYpQX3ykdIqagFaPp}
\textbf{Decorator Design pattern:} The system should support sending notifications via multiple channels (such as email or SMS) and allow for extensibility using the Decorator design pattern.
\item
\protect\phantomsection\label{jeyIGvyA0hnIKbH-wybI_}
\textbf{Singleton Design pattern:} The Singleton design pattern enforces that there should only be one shared instance of the \texttt{StockExchange} throughout the application.
\end{itemize}

\subsection{Object interactions}\label{oYuV_Td9YwffO0rV_G0PT}
\\
The system's dynamic behavior is captured through object collaboration during key activities:

\begin{itemize}
\item
\protect\phantomsection\label{7llBsW3vPB6sWqAIQfyUH}
\textbf{Placing an order:} The \texttt{Member} selects a stock and order details, which are sent to the \texttt{StockInventory} to verify availability and retrieve current prices.
\item
\protect\phantomsection\label{61jtQ0JO9fZJTD8ytoXHF}
\textbf{Order creation:} If sufficient stock (for selling) or funds (for buying) are available, the system creates an Order object (of the appropriate type).
\item
\protect\phantomsection\label{JJBls31ZtpyQ7pwxrs-u-}
\textbf{Stock adjustment:} The system deducts the specified quantity from the member's \texttt{StockPosition} or \texttt{StockInventory} for sales. For purchases, funds are deducted from the member's account.
\item
\protect\phantomsection\label{kI2hjlEtDeFSYEUSjIzta}
\textbf{Sending to exchange:} The \texttt{Order} details are forwarded to the \texttt{StockExchange}, which processes the transaction and returns an acknowledgment.
\item
\protect\phantomsection\label{dgBCV1Wup1xjmnm2mcwU9}
\textbf{Notification:} Upon successful execution or failure, a \texttt{Notification} (either \texttt{SmsNotification} or \texttt{EmailNotification}) is generated and sent to the \texttt{Member} to inform them of the result.
\item
\protect\phantomsection\label{EmwKePPWx8W1kTym94t0T}
\textbf{Transaction handling:} For deposits or withdrawals, a \texttt{DepositMoney} or \texttt{WithdrawMoney} object is created, the transaction is processed, and a corresponding notification is sent to the member.
\end{itemize}

\subsection{System workflow}\label{aTAge_INvDUrkvOJAejvt}
\\
The system supports several end-to-end workflows, which guide users and administrators through the main processes:

\begin{itemize}
\item
\protect\phantomsection\label{uv-GKpxJiEm_bmfeCkCY2}
\textbf{Stock purchase:} Member selects a stock to buy, enters order details, and submits. The system checks for sufficient deposit, deducts funds, creates an order, sends it to the stock exchange, and notifies the user of the outcome.
\item
\protect\phantomsection\label{s8rqiVxoa6m8sUfMlOCKL}
\textbf{Stock sale:} Member selects a stock to sell, configures order parameters, and submits. The system verifies stock availability, deducts the quantity, forwards the order to the exchange, and notifies the user.
\item
\protect\phantomsection\label{qGj23UMNKGeqvnomdZdU_}
\textbf{Account management:} Members or admins can create, update, or cancel accounts and reset passwords.
\item
\protect\phantomsection\label{9tnpsyEIwuVrlSuex8-fG}
\textbf{Watch list management:} Members can create, update, and view watchlists, tracking desired stocks and monitoring changes.
\item
\protect\phantomsection\label{-yJbrgYWinTViYG4MslEf}
\textbf{Deposits/Withdrawals:} Members deposit or withdraw funds via supported methods; the system processes transactions and sends notifications.
\item
\protect\phantomsection\label{4xI01yEuZ3YhEekf022y1}
\textbf{Order notifications:} The system automatically notifies members of order executions, cancellations, or failures, as well as deposit/withdrawal status.
\end{itemize}
