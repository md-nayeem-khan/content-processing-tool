\subsection{Design highlights}\label{4yZ-MXZNs_zxGjUCEZvVS}

\subsubsection{Design approach}\label{Wd3W3AOLYIP1Jb4e0RI0P}
\\
The system was designed using a bottom-up approach. This methodology identifies and designs the smallest, most granular components, such as the \texttt{Product} and \texttt{ProductReview} classes. These smaller components are then assembled into larger, more complex components like \texttt{ProductCategory} and \texttt{ShoppingCart}, continuing until the entire system is built.

\subsubsection{Design principles}\label{qek50DnYxoVHRVeUC9Kej}
\\
The design of the Amazon system effectively demonstrates the five SOLID principles, leading to a more maintainable, scalable, and robust architecture.

\begin{itemize}
\item
\protect\phantomsection\label{_v8adgq1iJJONOVDHrQOP}
\textbf{Single Responsibility principle (SRP):} Each class is designed to have one primary responsibility. For instance, the \texttt{Order} class manages order data and status, the \texttt{Payment} class handles payment processing, and the \texttt{Notification} class is solely responsible for sending notifications.
\item
\protect\phantomsection\label{ct1JxDZyJELPgx6BINLbW}
\textbf{Open/Closed principle (OCP):} The system is open to extension but closed for modification. This is evident in the use of abstract classes like \texttt{Payment} and \texttt{Notification}. New payment methods or notification types can be added by creating new subclasses without altering the existing, tested code.
\item
\protect\phantomsection\label{yQzHTYAru29Dmi6em6-Di}
\textbf{Liskov Substitution principle (LSP):} Subtypes can be substituted for their base types without altering the program's correctness. For example, the system can use a \texttt{Customer} object to function correctly, whether the concrete instance is an \texttt{AuthenticatedUser} or a \texttt{Guest}.
\item
\protect\phantomsection\label{1CyRDHjIhRe-goNatWuOP}
\textbf{Interface Segregation principle (ISP):} The \texttt{Search} interface is a clear example, providing only the methods relevant to searching. This ensures that client classes do not depend on methods they don't use, preventing bloated interfaces.
\item
\protect\phantomsection\label{JY5zBktr2qE-FjwxWFwGv}
\textbf{Dependency Inversion principle (DIP):} The system depends on abstractions rather than concrete implementations. The \texttt{Order} class interacts with the abstract \texttt{Payment} class, not directly with a specific \texttt{CreditCard} or \texttt{Cash} class, which decouples the components.
\end{itemize}

\subsubsection{Design patterns}\label{xRkOx9rO3BHYp4PAqRGFn}
\\
The system strategically employs several design patterns to solve common object-oriented design problems.

\begin{itemize}
\item
\protect\phantomsection\label{S6omLNPCRh8xb5xvCL4ZL}
\textbf{Factory Method pattern:} Used to create different types of accounts (\texttt{AuthenticatedUser}, \texttt{Guest}) without exposing the instantiation logic to the client.
\item
\protect\phantomsection\label{qUp3ypLcv2aghdY1sEK_b}
\textbf{Strategy pattern:} Implemented to handle different payment methods (\texttt{CreditCard}, \texttt{Cash}), allowing the payment algorithm to be selected at runtime.
\item
\protect\phantomsection\label{FKle3j4ppsfRyGkA_ZiZk}
\textbf{Composite pattern:} Used to treat individual \texttt{Product} objects and collections of products (\texttt{ProductCategory}, \texttt{ShoppingCart}) uniformly.
\item
\protect\phantomsection\label{_jmdvNSQNBVhHjnF_Jgex}
\textbf{Observer pattern:} Applied to notify customers automatically whenever their \texttt{Order} or \texttt{Shipment} status changes.
\item
\protect\phantomsection\label{MYdTMLq48D-KYhnYIomXN}
\textbf{Command pattern:} Used to encapsulate actions on objects like \texttt{Order} and \texttt{ShoppingCart} as distinct objects, enabling operations like \texttt{placeOrder()} and \texttt{addProduct()}.
\end{itemize}
