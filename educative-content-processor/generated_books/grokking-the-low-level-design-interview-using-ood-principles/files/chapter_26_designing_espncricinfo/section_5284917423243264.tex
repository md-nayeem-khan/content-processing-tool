\subsection{Design approach}\label{ofcRmrchLhX2hcxNiJizJ}
\\
The system is designed using a bottom-up approach, starting from the smallest units, such as \texttt{Ball}, \texttt{Run}, and \texttt{Wicket}, and then assembling them into higher-level entities, including \texttt{Over}, \texttt{Innings}, \texttt{Match}, and ultimately the full ESPNcricinfo platform.

\subsection{Design principles}\label{84e-UJSjAZE1xbwKKnUQn}
\\
The following SOLID principles are reflected in the system design:

\begin{itemize}
\item
\protect\phantomsection\label{lTw-EQRmGwxc5kSQY0D4S}
\textbf{Single Responsibility principle (SRP):} Each class is focused on a single responsibility, such as \texttt{Player} for cricketer details, \texttt{Ball} for per-delivery events, and \texttt{Commentary} for match commentary.
\item
\protect\phantomsection\label{3XVMykXFBVRqMh78FqJIx}
\textbf{Open/Closed principle (OCP):} New match types or statistics can be added by extending base classes (e.g., new child classes of \texttt{Match} or \texttt{Stat}).
\item
\protect\phantomsection\label{EvR_RVxQeTud51IhlQ8m3}
\textbf{Liskov Substitution principle (LSP):} Subclasses such as \texttt{ODI}, \texttt{Test}, and \texttt{T20} can be used wherever the base \texttt{Match} is expected.
\item
\protect\phantomsection\label{Ybl1QcMi5wr1kNyjld3Aj}
\textbf{Interface Segregation principle (ISP):} Responsibilities, such as adding commentary or assigning umpires, are split among appropriate classes and actors.
\item
\protect\phantomsection\label{0gib8bVaOFCbkyfGb7j0n}
\textbf{Dependency Inversion principle (DIP):} High-level modules (\texttt{Admin}, \texttt{Coach}) depend on abstractions, such as manipulating generic \texttt{Match} or \texttt{Team} objects, not concrete implementations.
\end{itemize}

\subsection{Design patterns}\label{jVUP5NlgQm4zfpat0z__f}
\\
The design pattern below is used in the ESPNcricinfo system:

\begin{itemize}
\item
\protect\phantomsection\label{mkNDwFGYiCaTPPQi3Nfpj}
\textbf{Factory pattern:} Used for dynamic creation of different match types, tournaments, and squads at runtime without exposing concrete classes.
\end{itemize}

\subsection{Object interactions}\label{Kd2gMrma29SZhsHNzgmpO}
\\
Key interactions between objects include:

\begin{itemize}
\item
\protect\phantomsection\label{AtxVMYFEu49SHh_CQXt-u}
\textbf{Adding a match:} The admin creates a new match (Test, ODI, or T20), adds teams and a stadium, and assigns umpires and commentators. Each action updates the related objects and relationships within the system.
\item
\protect\phantomsection\label{Hf2T9AI5XM9kdZmlgjIam}
\textbf{Recording a ball:} For each ball bowled, the system records its outcome (runs, wickets, commentary) by updating Ball, Run, Wicket, and Commentary objects within the current Over and Innings.
\item
\protect\phantomsection\label{McUD_aMuYPulPkTCRQnrG}
\textbf{Squad and playing 11 management:} Coaches manage \texttt{TournamentSquads} and choose the \texttt{Playing11} for each match, maintaining links between \texttt{Team}, \texttt{TournamentSquad}, and \texttt{Playing11} objects.
\end{itemize}

\subsection{System workflow}\label{GQ8rSViMyaCGe6Ez8h16C}
\\
These are the major user journeys and life cycle events in ESPNcricinfo:

\begin{itemize}
\item
\protect\phantomsection\label{SMtbl6-1uANXvqV_RGvzP}
\textbf{Tournament setup:} Admin adds tournaments and teams; coaches submit their squads.
\item
\protect\phantomsection\label{cFVsO6eDJT2ZrB4iKTtav}
\textbf{Live match execution:} Ball-by-ball events (deliveries, runs, wickets, commentary) are entered and stats are update in real time.
\item
\protect\phantomsection\label{Tjp5Dy_BgbK87tlSfLnZZ}
\textbf{Post-match processing:} Match outcomes are recorded, the points table is updated, and news or analysis can be added.
\item
\protect\phantomsection\label{jAeBkocQFL1ahYYN7Gio_}
\textbf{Statistics and history:} Users and admins can view stats and commentary for any player, team, match, or tournament, including historical data.
\end{itemize}
